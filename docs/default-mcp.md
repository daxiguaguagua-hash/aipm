# 默认 MCP 三件套

每个项目的标准 MCP 配置：graphify、gbrain、monorepo。三者定位不同，各司其职，构成一套完整的 AI 开发助手工具链。

## 各管什么

| MCP | 核心定位 | 管理内容 | 解决的问题 |
|-----|---------|---------|-----------|
| **graphify** | 代码理解 | 项目代码结构、依赖关系、调用链 | "这段代码的核心逻辑在哪？改了 A 会影响谁？" |
| **gbrain** | 长期记忆 | 用户知识、设计决策、会议记录、踩坑经验 | "我上周为什么这么设计？之前讨论过什么坑？" |
| **monorepo** | 仓库管理 | 多仓库文件定位、跨仓库引用 | "这个函数在哪个子仓库里？跨仓库依赖怎么找？" |

三个 MCP 覆盖三种完全不同的需求：写代码时的结构理解、长期设计决策记忆、多仓库间的跨库导航。

## 协作模式

AI 会根据问题类型自动选择合适的 MCP（或组合使用）。

### 写新功能

1. **gbrain** — 查历史决策："我之前有没有讨论过这个方案？"
2. **graphify** — 看代码结构："现有代码在哪？依赖哪些模块？"
3. **monorepo** — 跨仓库定位："相关服务在哪个子仓库？"
4. 综合信息后开始写代码

### 改动现有功能

1. **graphify** — 影响面分析："哪些地方调用了这个模块？改动会影响谁？"
2. **monorepo** — 跨仓库影响："相关服务分布在哪些仓库？需要同步改吗？"
3. **gbrain** — 避坑参考："之前为什么要这么设计？有没有踩过坑？"

### 排查问题

1. **graphify** — "状态机代码在哪里？流转逻辑是什么？"
2. **monorepo** — "这个状态的消费者在哪个仓库？"
3. **gbrain** — "之前有没有类似故障复盘？当时怎么修的？"

## 配置示例

```json
{
  "mcpServers": {
    "graphify": { … },
    "gbrain": { … },
    "monorepo": { … }
  }
}
```

## 初始化顺序

1. **monorepo MCP** — 如果确实在 monorepo 环境下工作
2. **graphify** — 对项目根目录跑 `graphify .` 构建图谱
3. **gbrain** — 导入已有的笔记和文档

## 优先级规则

在 AI 助手的配置（如 CLAUDE.md）中明确定义：

- 查找代码、理解结构 → 优先用 **graphify**
- 查询历史决策、设计理由 → 优先用 **gbrain**
- 跨仓库定位文件 → 优先用 **monorepo**
- 一个任务可能需要多个 MCP，按上述顺序依次查询

## 常见问题

### AI 不知道该用哪个 MCP

在 AI 助手的系统提示词中明确规则。也可给每个 MCP 配触发词，比如问代码加 `/code`，问历史加 `/brain`。

### MCP 之间信息重叠

以 graphify 为代码结构的**唯一事实源**，gbrain 只存设计决策和踩坑记录。gbrain 导入时排除自动生成的代码分析。

### 启动时加载慢

- graphify 按需启动
- monorepo MCP 配置为 lazy 模式（如支持）
- gbrain 常驻（最高频使用）

## 扩展方向

配置好这三件套后，可进一步添加：

- **GitHub MCP** — 搜 issue/PR，了解需求背景
- **Jira MCP** — 查任务上下文，知道"为什么要这么改"
